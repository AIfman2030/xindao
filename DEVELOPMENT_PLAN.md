## 第一阶段：核心聊天能力 (Week 1-2)

### F1.1: 真实 AI 对话引擎

**为什么先做这个（马斯克第一性原理）：**
> 聊天是产品的"发动机"。没有真实 AI，所有 UI 都是空壳。

**实现内容：**
- 构建 Web 后端服务（Node.js/Express）
- 对接 OpenAI API（GPT-4o Mini）
- 实现流式响应（打字机效果）
- 会话历史存储

**技术选型理由：**
- Express：轻量、快速上线
- GPT-4o Mini：成本效益最高（$0.15/1M tokens）
- 按 token 计费，用户自备 Key = 零 AI 成本

**成功标准：**
- 用户输入 → AI 回复 < 3 秒
- 支持多轮上下文对话
- Mock 模式自动降级

---

### F1.2: 行动卡智能生成

**为什么先做这个（乔布斯聚焦）：**
> 产品差异化不在聊天，在"行动"。聊完能落地，才是核心价值。

**实现内容：**
- AI 回复后自动提取行动项（3条以内）
- 行动卡持久化存储
- 点击完成/未完成切换

**提示词设计原则（参考 Naval Skill）：**
```
你是Naval Ravikant。用户刚刚聊了：[用户消息]
从中提取1-3个最值得立即执行的行动，不超过20字。
格式：每行一个行动项
```

**成功标准：**
- 每次对话后自动生成行动卡
- 行动卡可交互
- 数据本地持久化

---

## 第二阶段：用户系统 (Week 3)

### F2.1: 简单用户认证

**为什么（张一鸣数据思维）：**
> 没有用户体系，就没有数据积累。数据是未来的护城河。

**实现内容：**
- 匿名用户系统（设备ID）
- 昵称和一句话介绍
- 智者选择记录
- 本地优先 + 云端同步

**实现方式：**
- 设备指纹生成 UUID
- 云函数存储用户信息
- 轻量级，无密码登录

**成功标准：**
- 新用户 3 步完成注册
- 跨设备数据同步（可选）
- 用户数据可导出

---

### F2.2: 进度追踪系统

**为什么（游戏化设计）：**
> 打卡系统创造"连续行动"的游戏体验。连续天数是留存的关键。

**实现内容：**
- 今日行动列表
- 连续打卡天数统计
- 本周完成率
- 最近完成历史

**成功标准：**
- 打卡动作流畅
- 数据实时更新
- 连续打卡有视觉反馈

---

## 第三阶段：蒸馏功能 (Week 4-5)

### F3.1: 蒸馏 v1（预设模板）

**为什么（差异化核心）：**
> "蒸馏新智者"是产品最独特的卖点。用户可以创造自己的智囊团。

**实现内容：**
- 输入人名 + 选择标签
- 基于预设模板生成智者 prompt
- 添加到用户智囊团
- 本地存储

**马斯克风格实现：**
```
蒸馏 = 搜索 → 提炼 → 生成
每步可视化进度
用户可中途取消
```

**成功标准：**
- 3 分钟内完成蒸馏
- 生成的智者可正常对话
- 可创建多个自定义智者

---

### F3.2: 智者切换与切换

**实现内容：**
- 智者列表展示
- 一键切换当前智者
- 对话历史按智者隔离

**成功标准：**
- 切换流畅
- 历史不丢失
- 视觉反馈清晰

---

## 第四阶段：推送与提醒 (Week 6)

### F4.1: 本地通知提醒

**为什么（张一鸣）：**
> "知道"不等于"做到"。提醒是连接"聊天"到"行动"的桥梁。

**实现内容：**
- 行动卡设置提醒时间
- 浏览器通知权限
- 定时弹窗提醒
- 点击跳转行动卡

**成功标准：**
- 通知准时送达
- 点击率可追踪
- 用户可管理通知时间

---

### F4.2: 打卡激励系统

**实现内容：**
- 连续 3/7/30 天徽章
- 完成率数据可视化
- 鼓励文案（智者也来祝贺）

**乔布斯风格：**
> 仪式感很重要。每次打卡完成，智者应该说一句话。

---

## 第五阶段：数据与迭代 (Week 7-8)

### F5.1: 埋点与数据收集

**实现内容：**
- 功能使用埋点（匿名）
- 核心指标：DAU、对话数、打卡率
- 用户旅程漏斗

**张一鸣数据框架：**
```
北极星指标：连续7天打卡的用户的留存率
次级指标：每日对话数、平均对话轮次
```

---

### F5.2: A/B 测试框架

**实现内容：**
- 随机分流
- 不同智者 prompt 效果对比
- 行动卡生成策略对比
